# ADR-001: Вибір мови програмування і HTTP-фреймворку

**Статус:** Прийнято

**Дата:** 2026-05-08

**Автор:** Yurii Levchenko

## Контекст

Кейс SE School 6.0 вимагає реалізації HTTP API із такими обмеженнями:

- Дозволені мови: Go, Node.js, PHP
- Заборонені "товсті" фреймворки: Nest.js (Node.js), Revel і Fx (Go), Laravel (PHP)
- Дозволені тонкі фреймворки: Gin / Chi / net/http (Go), Express / Fastify (Node.js), Slim (PHP)
- Сервіс повинен бути моноліт (API + Scanner + Notifier у одному процесі)
- Мій основний бекграунд — Python/Django; обрана мова буде новою для мене

Додатковий контекст: Genesis активно використовує Go у своїх продуктах (зокрема Solidgate, TENTENS Tech), тому вибір Go дає кращу узгодженість зі стеком потенційного роботодавця і для розширення знань в Back-end розробці, враховуючи що Go - протилежність Python.

## Розглянуті варіанти

### 1. Go + Gin
- **Плюси:** найбільша екосистема серед Go-фреймворків, велика кількість туторіалів і прикладів, зручний JSON binding із валідацією, багато готових middleware, голанг-комʼюніті активне.
- **Мінуси:** використовує власний `gin.Context` замість чистого `net/http.Handler` — менш ідіоматичний ніж Chi; невелика "framework lock-in" небезпека.

### 2. Go + Chi
- **Плюси:** ідіоматичний, повністю сумісний із `net/http`, легкий, гарна композиція middleware, мінімалістичний.
- **Мінуси:** потрібно більше boilerplate для типових задач (наприклад, JSON binding треба писати самому).

### 3. Go + net/http (stdlib only)
- **Плюси:** нуль зовнішніх залежностей, повний контроль.
- **Мінуси:** забагато boilerplate для маршрутизації з параметрами в шляху, ручна валідація, ручний JSON binding.

### 4. Node.js + Express
- **Плюси:** найшвидший ramp-up для людини з Python (динамічна типізація), велика екосистема npm.
- **Мінуси:** менш узгоджено зі стеком Genesis; немає переваг статичної типізації; гірша performance і вища пам'ять за ідентичну роботу.

### 5. PHP + Slim
- **Плюси:** простий, схожий на легкі Python-фреймворки.
- **Мінуси:** PHP менше використовується, найменш привабливий варіант для довгострокового напрямку.

## Прийняте рішення

Обрано **Go + Gin**.

Критерії вибору в порядку пріоритету:
1. Узгодженість зі стеком Genesis → виключає PHP.
2. Static typing і compiled binary → виключає Node.js.
3. Найкоротший час до робочого прототипу при першому досвіді з Go → виключає чистий net/http.
4. Між Gin і Chi обрано Gin за ширшу екосистему і документацію.

## Наслідки

### Позитивні
- Single binary deployment — простий Docker-image ~15 МБ
- Goroutines дозволяють реалізувати Scanner inline без зовнішнього scheduler (див. ADR-0004)
- Static typing ловить помилки на етапі компіляції — менше runtime-сюрпризів
- Gin's binding автоматично валідує JSON-body через struct tags
- Перенос знань на потенційні Genesis-проєкти

### Негативні
- Власний `gin.Context` створює невелику framework-залежність (проте handlers тонкі і ізольовані від бізнес-логіки)
- Порівняно з Node.js для Python-розробника важче перейти.
- Менше готових prom/middleware за Chi (покрито через прямі бібліотеки типу `prometheus/client_golang`)
