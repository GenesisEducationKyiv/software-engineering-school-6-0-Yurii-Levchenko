# ADR-002: Архітектура — моноліт із пошаровим розділенням

**Статус:** Прийнято

**Дата:** 2026-05-08

**Автор:** Yurii Levchenko

## Контекст

Сервіс має реалізувати три відповідальності в одному застосунку:

- **API** — HTTP-ендпойнти для управління підписками
- **Scanner** — фоновий процес, що перевіряє GitHub на нові релізи
- **Notifier** — надсилання email-сповіщень

Завдання вимагає реалізації монолітом — мікросервіси заборонені на цьому етапі.

Хоча мікросервіси заборонені формально, всередині моноліта потрібна структура, яка:
- Забезпечує тестованість бізнес-логіки в ізоляції
- Дозволяє чітко локалізувати помилки
- Не блокує можливе майбутнє розщеплення на сервіси

## Розглянуті варіанти

### 1. Flat structure (всі файли у корені пакета)
- **Плюси:** швидкий старт.
- **Мінуси:** усе в одному пакеті, неможливо тестувати без БД, заплутано в багатьох файлах.

### 2. Layered monolith (handler → service → repository)
- **Плюси:** чіткий поділ HTTP-шару, бізнес-логіки і доступу до даних. Знайомий підхід для розробників із Django (Views/Services/Models або MVC/T). Легко мокати залежності.
- **Мінуси:** дещо більше boilerplate (інтерфейси, dependency injection в `main.go`).

### 3. Hexagonal Architecture (Ports & Adapters)
- **Плюси:** ще кращий поділ домену від інфраструктури; повна незалежність бізнес-логіки.
- **Мінуси:** надмірно складно для проєкту такого масштабу (2 таблиці, 4 ендпойнти); багато додаткових абстракцій без реальної цінності.

### 4. Domain-Driven Design with bounded contexts
- **Плюси:** масштабований підхід для великих доменів.
- **Мінуси:** домен надто простий (одне поняття — Subscription)

## Прийняте рішення

Обрано **Layered monolith** із такими шарами:

```
HTTP Router (Gin)
      ↓
Handlers (parse request, return response)
      ↓
Service layer (business logic, validation, orchestration)
      ↓
Repository layer (SQL queries via sqlx)
```

Паралельно з API-стеком — окрема горутина-сканер, яка використовує ті самі Service- і Repository-компоненти.

Структура пакетів:

```
internal/
├── handler/      — HTTP-хендлери
├── service/      — бізнес-логіка + кастомні error-типи
├── repository/   — SQL-запити
├── github/       — клієнт GitHub API + кешований wrapper
├── scanner/      — фоновий Goroutine
├── notifier/     — SMTP-клієнт
├── cache/        — Redis-обгортка
├── metrics/      — Prometheus-метрики
├── middleware/   — API-key auth
├── model/        — структури даних
└── config/       — завантаження env
```

## Наслідки

### Позитивні
- **Тестованість:** Service-шар тестується з моками без БД (13 юніт-тестів, 82% coverage)
- **Зрозумілість:** новий розробник одразу розуміє "де що лежить" за іменами папок
- **Локалізація змін:** зміна формату HTTP-відповіді чіпає лише handler; зміна логіки збереження — лише repository
- **Розширюваність:** додавання нового ендпойнта = новий метод у service + новий handler
- **Майбутня міграція:** при потребі винести scanner у окремий сервіс — потрібно лише обернути його у власний `main.go` з тими ж залежностями

### Негативні
- **Boilerplate:** інтерфейси для тестів, явний Dependency Injection у `main.go` — більше коду ніж у flat-підході
- **Indirection:** відслідковування якщо щось відбулось вимагає переходу через 3 шари (handler → service → repository)
- **Спокуса додавати нові шари**
