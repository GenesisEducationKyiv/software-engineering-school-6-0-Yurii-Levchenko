# ADR-004: Сканер релізів — внутрішня goroutine замість зовнішнього шедулера

**Статус:** Прийнято

**Дата:** 2026-05-08

**Автор:** Yurii Levchenko

## Контекст

Сервіс повинен періодично (кожні 5 хвилин) перевіряти GitHub API на нові релізи для всіх активних підписок:

- Виявити, які репо мають хоча б одного підтвердженого підписника
- Для кожного — викликати GitHub API за `latest_release`
- Порівняти з збереженим `last_seen_tag`
- Якщо тег новий — оновити стан і надіслати email усім підписникам

Завдання вимагає моноліт — все має жити в одному процесі. Незважаючи на це, треба обрати спосіб реалізації періодичного запуску всередині процесу.

Контекст: я з Python/Django, де стандартний підхід для подібного — Celery worker з Redis. У Go конкурентність вбудована в мову (goroutines, channels, select), що відкриває простіші варіанти.

## Розглянуті варіанти

### 1. In-process goroutine з time.Ticker
- **Плюси:** Нуль додаткової інфраструктури. Спільний state з API (БД, Redis, GitHub-клієнт). Пряма передача `context.Context` — graceful shutdown працює end-to-end. Найпростіший дебагінг — все в одному процесі і одних логах.
- **Мінуси:** При падінні сервісу — сканер теж падає (mitigated через Docker `restart: on-failure`). Не масштабується горизонтально без додаткової координації (якщо запустимо 2 інстанси — обидва скануватимуть).

### 2. OS cron у тому ж Docker-контейнері
- **Плюси:** Знайомий патерн.
- **Мінуси:** Cron живе окремо від основного процесу — потрібен supervisor щоб тримати обидва процеси. Складніший Dockerfile. Cron викликає бінарник кожен раз — нема shared state, потрібно щоразу заново ініціалізувати DB-pool, Redis-клієнт, тоді логи будуть теж розкидані.

### 3. Окремий worker-процес (Celery-style)
- **Плюси:** Схоже на реалізацію в Python. Чітке розділення ролей API і worker.
- **Мінуси:** Порушує вимогу моноліта. Потребує message broker  типу Redis чи RabbitMQ. Завелика інфраструктурна складність для завдання.


## Прийняте рішення

Обрано **In-process goroutine з time.Ticker і context.Context**.

Реалізація:
```go
func (s *Scanner) Start(ctx context.Context) {
    s.scan(ctx) // негайний запуск при старті

    ticker := time.NewTicker(s.interval)
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return // graceful shutdown
        case <-ticker.C:
            s.scan(ctx)
        }
    }
}
```

Сканер запускається з `main.go` через `go releaseScanner.Start(ctx)`, де `ctx` — головний контекст застосунку, який скасовується при SIGTERM/SIGINT.

## Наслідки

### Позитивні
- **Нуль додаткової інфраструктури:** ніяких брокерів, supervisor-ів, окремих контейнерів
- **Shared state:** сканер користується тими самими інстансами Repository, GitHubClient, Notifier
- **Graceful shutdown працює end-to-end:** SIGTERM → main викликає `cancel()` → scanner бачить `<-ctx.Done()` → повертається; HTTP-запити до GitHub теж скасовуються (див. ADR-006)
- **Простота дебагінгу:** усі логи в одному потоці; легко відстежити повний цикл
- **Ідіоматично для Go:** використовує саме ті фічі, які роблять мову популярною

### Негативні
- **Залежить від life cycle app:** падіння процесу => зупинка сканера. Але компенсується через `restart: on-failure` в docker-compose
- **Не масштабується горизонтально:** запуск 2+ інстансів призведе до дублікатних повідомлень і race conditions при оновленні `last_seen_tag`. У майбутньому можна додати distributed lock (Redis-lock або Postgres advisory lock) перед циклом сканера
- **Single point of failure:** немає built-in retry за 5-хвилинний інтервал — якщо цикл провалиться через мережеву помилку, наступна спроба буде через 5 хв. Прийнятно для цього use case бо релізи як правило не виходять кожну секунду
